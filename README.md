For this project, I worked with DriverPass, a company that wanted to help students better prepare for their driving tests. The client’s goal was to create a system that combined online practice exams with scheduling for in-person driving lessons. The system needed to support multiple users like students, instructors, and administrators, while also handling things like account management, scheduling, payments, and progress tracking. Overall, the goal was to design a system that made it easier for students to study and book driving time all in one place.

One thing I think I did really well was translating the client’s requirements into a clear and organized system design. I made sure the documents were easy to follow and covered both the business side and the technical side of the system. I also focused on making my diagrams and explanations straightforward so someone else could pick them up and understand how the system works without confusion.

If I could go back and revise one part of my work, I would improve the level of detail in my UML diagrams. While they captured the main functionality, I think I could have added more depth, especially in areas like relationships between objects or edge cases in the system. Adding that extra detail would make the design even more useful for developers who might build the system later.

When interpreting the user’s needs, I focused on what the client was trying to accomplish and how different users would interact with the system. I used the interview transcript to identify key features like scheduling lessons, accessing practice tests, and managing user accounts, then built those directly into the design. Considering the user’s needs is important because if the system doesn’t actually solve their problems or is difficult to use, then it fails no matter how well it’s built technically.

When I approach designing software, I like to start by fully understanding the problem and breaking it down into smaller pieces. From there, I focus on gathering requirements, creating diagrams like use case and class diagrams, and thinking through how users will interact with the system step by step. In the future, I would continue using these techniques along with iterative design, getting feedback early, and refining the system as needed. This helps make sure the final product is both functional and user-friendly.
